# CWE-502: Deserialization of Untrusted Data

## Description

The product deserializes untrusted data without sufficiently ensuring that the resulting data will be valid. **Serialization** is the process of packaging program-internal object data for storage or transfer; **deserialization** is the reconstruction of the original object from that data. When an application reconstructs an object from an attacker-controlled stream, it may inadvertently execute dangerous methods or transition into an unexpected state.

## Why This Matters (Security Impact)

Deserialization vulnerabilities are often "silent" but lead to total system compromise.

* **Arbitrary Code Execution (ACE):** Attackers use **gadget chains**—sequences of existing library methods that, when invoked during the reconstruction process, self-execute to perform unauthorized actions, such as spawning a shell.
* **Integrity Violation:** Attackers can modify object attributes directly, bypassing accessor functions (getters/setters) and security logic.
* **Unexpected State:** The application may be forced into logic flows that were assumed to be unreachable, potentially bypassing authentication.

## Common Consequences

* **Execute Unauthorized Code:** Triggering malicious logic during the object reconstruction phase.
* **Modify Application Data:** Altering internal variables that the application assumes are safe.
* **Denial of Service (DoS):** Creating payloads that cause infinite loops or massive CPU consumption during parsing (e.g., "Billion Laughs" style attacks or sentry-based string failures).

## Likelihood of Exploit

**Medium**
Exploitation requires "gadgets" (specific classes/methods) to be present in the application's classpath or environment, but many common libraries contain known gadgets.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Tools trace data flow from untrusted sources (network/disk) to deserialization "sinks" (e.g., `readObject()` in Java, `unserialize()` in PHP, `pickle.load()` in Python). (Effectiveness: High).
* **Manual Review:** Identifying where the application accepts serialized objects from the client (e.g., cookies, hidden form fields, or API parameters).

## Potential Mitigations

### Architecture and Design

* **Integrity Checks:** Use digital signatures (e.g., **HMAC**) to "seal" serialized data. This ensures that the data has not been modified by an untrusted party before deserialization.
* **Populate New Objects:** Instead of directly deserializing an untrusted stream into an object, extract the raw data and use it to populate a fresh, manually-created object. This forces the data through standard input validation.
* **Encryption:** Encrypt the serialized data to prevent unauthorized viewing or tampering, though this is often only a partial "client-side" defense.

### Implementation

* **Allowlisting Classes:** Explicitly define which classes are acceptable for deserialization. Reject any type not on the list to prevent the use of unexpected gadgets.
* **Transient Fields:** Mark sensitive, time-based, or environment-specific fields as `transient` to prevent them from being serialized and later used improperly.
* **Finalize to Prevent:** Explicitly define a `final object()` method or similar constructs to prevent classes from being improperly extended or used in deserialization attacks.

### Operation

* **Application Firewall:** Use a WAF to detect and block common deserialization payload patterns, especially those targeting known library gadgets. (Effectiveness: Moderate).

## Simplified Example

```python
# VULNERABLE: Python's pickle is inherently unsafe with untrusted data
import pickle
def load_user_settings(user_data):
    return pickle.loads(user_data) # Attacker can inject a malicious object

# SECURE: Use a data-only format like JSON
import json
def load_user_settings(user_data):
    # JSON only allows data, not executable code/objects
    return json.loads(user_data) 

```
