# CWE-284: Improper Access Control

## Description

The product does not restrict or incorrectly restricts access to a resource from an unauthorized actor. Access control is a broad "umbrella" category that encompasses the protection mechanisms used to steer actors away from resources they should not be able to reach.

## The Three Pillars of Access Control

When any of these mechanisms fail, the security of the product is compromised:

1. **Authentication:** Proving the identity of an actor (e.g., "Are you who you say you are?").
2. **Authorization:** Ensuring a given actor can access a specific resource (e.g., "Do you have permission to see this file?").
3. **Accountability:** Tracking the activities performed (e.g., "Who deleted this record and when?").

## Why This Matters (Security Impact)

Failure to implement proper access control allows attackers to:

* **Gain Privileges:** Elevate their status to administrator or root.
* **Sensitive Data Exposure:** Read files, passwords, or personal user data.
* **Execute Commands:** Run unauthorized code or system instructions.
* **Evade Detection:** Perform malicious actions without being logged or tracked.

## Two Types of Access Control Failures

Access control weaknesses generally fall into one of two categories:

### 1. Specification Errors

The privileges or permissions are explicitly defined incorrectly.

* **Example:** Setting a sensitive configuration file to be "world-writable."
* **Example:** Assigning "Administrator" roles to a "Guest" account.

### 2. Enforcement Errors

The software contains bugs that prevent it from following the specified policy.

* **Example:** A system has a policy that only admins can delete users, but the "Delete" API endpoint doesn't actually check the user's role before executing.
* **Example:** Allowing a user to modify their own privilege level in a web request.

## Potential Mitigations

### Architecture and Design

* **Separation of Privilege:** Compartmentalize the system into "safe" areas with clear **trust boundaries**. Do not let sensitive data cross these boundaries without strict checks.
* **Principle of Least Privilege:** Users and processes should operate using the minimum privileges necessary to complete their tasks. Drop unnecessary privileges as soon as they are no longer needed.
* **Trust Zone Management:** Explicitly define and manage which parts of the software are trusted and which interface with untrusted external actors.

### Operation

* **Privilege Management:** Carefully audit and manage the settings and handling of user rights and resource permissions.
* **Default Deny:** Ensure that the system defaults to "no access" unless a specific permission is granted.

## Simplified Example

```python
# VULNERABLE: The 'Enforcement' is missing. 
# It assumes only admins see the link, but the function doesn't check.
@app.route('/admin/delete_db')
def delete_database():
    db.drop_all() 
    return "Database deleted."

# SECURE: The mechanism enforces the specification.
@app.route('/admin/delete_db')
def delete_database():
    user = get_current_user()
    # Explicitly check authorization before accessing the resource
    if user.is_authenticated and user.has_role('ADMIN'):
        db.drop_all()
        log_action(user, "Deleted Database") # Accountability
        return "Database deleted."
    else:
        abort(403) # Forbidden

```
