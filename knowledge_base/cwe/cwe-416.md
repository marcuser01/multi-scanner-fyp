# CWE-416: Use After Free

## Description

The product reuses or references a memory address after it has been explicitly freed. When memory is freed, it is returned to the system allocator; if the original pointer (a "dangling pointer") is used again, it may point to memory that has since been reallocated to a different part of the program. This leads to undefined behavior, as the original pointer now operates on data "owned" by a new allocation.

## Why This Matters (Security Impact)

Use After Free (UAF) is a premier exploitation vector in C/C++ applications (like browsers and kernels). It allows:

* **Write-What-Where Primitives:** Attackers can manipulate the new data via the old pointer to overwrite critical structures.
* **Arbitrary Code Execution (ACE):** If the reallocated memory holds a C++ object, an attacker can overwrite its virtual function table (vtable) to redirect execution to a malicious payload.
* **Information Disclosure:** Reading from the freed pointer can leak sensitive data from the newly allocated block.

## Common Consequences

* **Modify Memory:** Corrupting valid data in a newly allocated block, leading to logic errors.
* **Execute Unauthorized Code:** Redirecting function pointers or return addresses to attacker-controlled shellcode.
* **Denial of Service (DoS):** Triggering crashes during chunk consolidation or when the system attempts to use corrupted memory metadata.

## Likelihood of Exploit

**High**
UAF vulnerabilities are highly reliable for exploitation and are frequently used in "Zero-Day" chains to bypass modern sandboxes and OS-level protections.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Tools like **Semgrep** or specialized C analyzers track the lifecycle of pointers to identify references following a `free()` or `delete` call. (Effectiveness: High).
* **Automated Dynamic Analysis (ASan):** **AddressSanitizer** is the industry standard for detecting UAF at runtime by "poisoning" freed memory and monitoring for unauthorized access. (Effectiveness: Moderate/High).
* **Fuzzing:** Generating complex execution paths to trigger the specific timing required for a UAF condition. (Effectiveness: High).

## Potential Mitigations

* **Language Selection:** Use languages with automatic garbage collection (Java, Python, Go) or strict ownership models (**Rust**) that prevent manual memory management errors by design.
* **Nullify Pointers:** Immediately set pointers to `NULL` after calling `free()`. While this may still cause a crash (NULL dereference), it prevents the pointer from being used for code execution.
* **Smart Pointers:** In C++, use `std::unique_ptr` or `std::shared_ptr` to automate memory deallocation and reduce the risk of manual management mistakes.
* **Consistent Lifetime Management:** Use clear ownership patterns where only one component is responsible for the allocation and deallocation of a specific resource.

## Simplified Example

```cpp
// VULNERABLE: Pointer is used after free
char* data = (char*)malloc(SIZE);
free(data);
// ... some other code reallocates this memory ...
strncpy(data, "malicious", SIZE); // USE AFTER FREE

// SECURE: Pointer is nullified to prevent reuse
char* data = (char*)malloc(SIZE);
if (data) {
    free(data);
    data = NULL; // Prevents accidental reuse
}

if (data != NULL) {
    strncpy(data, "safe", SIZE);
}

```
