# CWE-200: Exposure of Sensitive Information to an Unauthorized Actor

## Description

The product exposes sensitive information to an actor that is not explicitly authorized to have access to that information. This occurs when the application fails to protect data confidentiality, either by explicitly including sensitive data in out-going messages or by failing to prevent unauthorized access to resources where that data is stored.

## What Constitutes "Sensitive Information"?

The definition of sensitivity varies by context and stakeholder (users, admins, developers), but generally includes:

* **PII (Personally Identifiable Information):** Financial data, health records, messages, and geographic location.
* **System & Environment Details:** Operating system versions, installed packages, or full local file paths.
* **Technical Metadata:** Connection logs, message headers, and internal stack traces.
* **Business Intelligence:** Intellectual property and network configuration.
* **Indirect Info:** Discrepancies in timing or error responses that reveal internal states (Side Channels).

## How Exposure Happens

1. **Explicit but Unscrubbed:** The code intentionally sends a message but forgets to "sanitize" it (e.g., sending a full user object to the frontend including the password hash).
2. **Indirect Leakage:** A mistake elsewhere reveals data (e.g., a database error displays the raw SQL query and server path in the browser).
3. **Unintentional Access:** Sensitive resources are managed correctly but made accessible due to a different weakness (e.g., a backup file left in a public web directory).

## Why This Matters (Security Impact)

* **Loss of Confidentiality:** The primary impact is the unauthorized reading of application data.
* **Reconnaissance:** Attackers use leaked system details (OS version, software versions) to tailor more advanced attacks.
* **Identity Theft:** Exposure of PII can lead to fraud or account takeovers.

## Likelihood of Exploit

**High**
Information leaks are frequently exploited because they are often easy to find—sometimes appearing directly in browser URLs or error pages—and provide the "blueprint" for further attacks.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Highly effective at tracing data flow from sensitive "sources" (like a database) to public "sinks" (like an HTTP response). (Effectiveness: High).
* **Web Application Scanners (DAST):** Scanners crawl the app looking for sensitive keywords (e.g., "root:", "SQL Syntax Error") in responses. (Effectiveness: High).
* **Manual Source Code Review:** Critical for finding business-specific leaks that automated tools might not recognize as "sensitive." (Effectiveness: High).
* **Fuzzing:** Using malformed input to trigger error messages that might reveal internal paths or logic. (Effectiveness: Partial).

## Potential Mitigations

### Architecture and Design

* **Separation of Privilege & Compartmentalization:** Divide the system into "safe" areas. Ensure sensitive data never leaves the trust boundary without being explicitly sanitized.
* **Principle of Least Privilege:** Only grant access to sensitive data at the specific moment it is needed, and drop those privileges immediately after.
* **Trust Boundaries:** Ambiguously draw boundaries between internal processes and external interfaces, scrubbing all data that crosses to the outside.

### Implementation

* **Generic Error Messages:** Ensure that application errors provide only the minimal information necessary to the user (e.g., "An internal error occurred") while logging the technical details securely on the server.
* **Data Masking:** Scrub or mask sensitive fields (like credit card numbers or internal IDs) before they are sent to the client-side or logged.

## Simplified Example

```python
# VULNERABLE: Exposes the entire user object, including sensitive fields
@app.route('/api/user/<id>')
def get_user(id):
    user = db.query_user(id)
    return jsonify(user.__dict__) # Might leak 'password_hash' or 'internal_notes'

# SECURE: Explicitly defines what data is safe to expose
@app.route('/api/user/<id>')
def get_user(id):
    user = db.query_user(id)
    safe_data = {
        "username": user.username,
        "display_name": user.display_name,
        "join_date": user.join_date
    }
    return jsonify(safe_data)

```
