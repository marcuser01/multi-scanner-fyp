# CWE-770: Allocation of Resources Without Limits or Throttling

## Description

The product allocates a reusable resource or group of resources on behalf of an actor without imposing any intended restrictions on the size or number of resources that can be allocated. By making rapid or oversized requests, an attacker can consume all available resources, preventing legitimate users from accessing the system.

## Why This Matters (Security Impact)

* **Denial of Service (DoS):** This is the primary impact. Attackers can crash an application or make it unresponsive by exhausting CPU, memory, disk space, or network bandwidth.
* **Cascading Failures:** An uncontrolled failure in one component (e.g., a database connection pool) can cause other downstream services to fail.
* **Economic Denial of Sustainability (EDoS):** In cloud environments, an attacker can scale resources infinitely, leading to massive financial costs for the victim rather than a technical crash.
* **Identity Impersonation:** Throttling mechanisms that block specific users can be abused by attackers to "lock out" valid users by impersonating them and intentionally triggering the limit.

## Common Consequences

* **DoS: Resource Consumption:** Excessive use of CPU, Memory, or File Descriptors.
* **System Unstability:** Causing the program to exit, restart, or enter an infinite loop.

## Likelihood of Exploit

**High**
Exploiting a lack of throttling is often trivial, requiring only a simple script to send a high volume of requests or a single request for a massive resource (e.g., a multi-gigabyte file download).

## How This Is Typically Detected

* **Fuzzing:** Opportunistic detection. If a fuzzer sends many test cases without a restart and the app eventually crashes, it often points to a resource leak or unlimited allocation. (Effectiveness: Opportunistic).
* **Automated Dynamic Analysis:** Stress testing by generating a high volume of requests in a short timeframe. (Effectiveness: High).
* **Automated Static Analysis (SAST):** Can find missing "release" calls for files or sockets, but struggles to identify missing business-logic limits (e.g., "max 5 posts per day"). (Effectiveness: Limited).
* **Manual Review:** Necessary to ensure business rules have corresponding technical limits. (Effectiveness: Moderate).

## Potential Mitigations

### Architecture and Design

* **Throttling & Rate Limiting:** Track the rate of requests from specific users or IP addresses and block those that exceed a defined threshold.
* **Per-User Quotas:** Set specific limits on how many resources (memory, storage, connections) a single unprivileged user can consume.
* **Graceful Failure (Fail Closed):** Ensure that if the system runs out of resources, it fails into a safe posture rather than leaving the system in an undefined state.
* **Centralized Authentication:** Implementing strong access control makes it easier to track and block the specific actor responsible for the abuse.

### Implementation

* **Input Validation:** Reject any input that specifies a resource size (e.g., `length`, `count`) that exceeds established safety limits.
* **Server-Side Checks:** Never trust a client to limit its own resource usage. Duplicate all limits on the server side.
* **Resource Monitoring:** Use environment settings like `setrlimit()` (POSIX) to enforce OS-level caps on what the process can claim.

### Operation

* **Caching:** Minimize resource expenditure by caching expensive result sets, reducing the need for repeated database or CPU-intensive lookups.

## Simplified Example

```python
# VULNERABLE: Allows the user to specify how many records to generate in memory
@app.route('/generate_report')
def generate_report():
    count = int(request.args.get('count'))
    # If count is 1,000,000,000, the server will run out of memory (OOM)
    data = [expensive_operation() for _ in range(count)]
    return jsonify(data)

# SECURE: Implements a maximum limit (Quotas) and throttling
@app.route('/generate_report')
@limiter.limit("5 per minute") # Rate limiting
def generate_report():
    count = int(request.args.get('count'))
    
    # 1. Enforce a hard maximum (Input Validation)
    MAX_RECORDS = 100
    safe_count = min(count, MAX_RECORDS)
    
    # 2. Allocate within safe bounds
    data = [expensive_operation() for _ in range(safe_count)]
    return jsonify(data)

```
