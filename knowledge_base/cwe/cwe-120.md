# CWE-120: Buffer Copy without Checking Size of Input ('Classic Buffer Overflow')

## Description

The product copies an input buffer to an output buffer without verifying that the size of the input is less than the size of the destination. This "classic" overflow occurs when more data is placed in a buffer than it can hold, resulting in data being written to memory areas outside the intended boundaries.

## Why This Matters (Security Impact)

The existence of a classic overflow strongly suggests a lack of basic security protections. It is the "Public Enemy #1" of software security because:

* **Arbitrary Code Execution (ACE):** Attackers can overwrite control data (like return addresses or function pointers) to redirect the execution flow to malicious code.
* **System Subversion:** A successful exploit can subvert nearly any other security service or implicit policy.
* **Total Compromise:** By overwriting adjacent variables, attackers can change the software's state, escalate privileges, or bypass authentication.

## Common Consequences

* **Modify Memory:** Corrupting valid data or control structures.
* **Execute Unauthorized Code:** Running arbitrary commands with the application's privileges.
* **Denial of Service (DoS):** Triggering segmentation faults, crashes, or infinite loops that consume CPU resources.

## Likelihood of Exploit

**High**
Despite decades of awareness, classic overflows remain common in legacy C/C++ code and are high-value targets for attackers.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Highly effective. Modern tools use data flow and constraint-based techniques to find unbounded copy operations (e.g., `gets()`, `strcpy()`). (Effectiveness: High).
* **Automated Dynamic Analysis (Fuzzing):** Sending massive, diverse inputs to trigger crashes or memory corruption. (Effectiveness: Moderate).
* **AddressSanitizer (ASan):** Runtime error-checking that flags illegal memory writes during development. (Effectiveness: Moderate).
* **Design Review:** Identifying the use of unsafe, unbounded functions in the codebase. (Effectiveness: High).

## Potential Mitigations

* **Language Selection:** Use memory-safe languages like Java, Perl, or C# that perform automatic memory management.
* **Vetted Libraries:** Use safer alternatives to standard C functions, such as **Strsafe.h** (Microsoft) or the **Safe C String Library (SafeStr)**.
* **Replace Unbounded Functions:** Substitute `strcpy` with `strncpy`, `strcat` with `strncat`, and `gets` with `fgets`.
* *Note: Be careful of off-by-one errors and missing NULL terminators when using bounded functions.*


* **Compiler Hardening:**
* **Stack Canaries:** Use flags like `/GS` (Visual Studio) or `StackGuard` (GCC) to detect stack frame corruption.
* **ASLR:** Randomize memory layout to prevent predictable jumps to malicious code.
* **DEP/NX:** Mark data segments as non-executable to prevent the CPU from running code injected into buffers.


* **Input Validation:** Assume all input is malicious. Truncate strings to reasonable lengths and use "accept known good" validation strategies.
* **Principle of Least Privilege:** Run the process with the lowest possible privileges to limit the damage of a successful exploit.

## Simplified Example

```c
// VULNERABLE: No check on the length of 'src'
void copy_data(char *src) {
    char dest[10];
    strcpy(dest, src); // If src > 10 chars, memory is corrupted
}

// SECURE: Length is restricted and null-termination is ensured
void copy_data(char *src) {
    char dest[10];
    strncpy(dest, src, sizeof(dest) - 1);
    dest[sizeof(dest) - 1] = '\0';
}

```
