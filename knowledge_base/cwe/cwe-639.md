# CWE-639: Authorization Bypass Through User-Controlled Key

## Description

The system's authorization functionality does not prevent one user from gaining access to another user's data or record by modifying the key value identifying the data. While the user may be authenticated, the system fails to verify if that specific user has the right to access the specific record identified by the key they provided.

## How the Attack Occurs

The vulnerability exists when data retrieval is based on a key (ID) that is under the user's control. Attackers identify these keys in:

* **URL Parameters:** `example.com/view_profile?user_id=123`
* **Hidden Form Fields:** `<input type="hidden" name="account_id" value="987">`
* **Cookies:** Unencrypted or plain-text variables stored in the browser.
* **Sequential IDs:** If IDs are `1001`, `1002`, `1003`, an attacker can easily iterate (increment/decrement) through them to discover other records.

## Why This Matters (Security Impact)

* **Horizontal Privilege Escalation:** An authenticated user accesses data belonging to another user of the same privilege level (e.g., viewing another customer's bank statement).
* **Vertical Privilege Escalation:** If the controlled key is a status flag (e.g., `is_admin=0`), the attacker may modify it to gain administrative access.
* **Bypass Protection Mechanisms:** Standard access control checks are circumvented because the system trusts the "key" provided by the client without verifying ownership.

## Common Consequences

* **Bypass Access Control:** Gaining unauthorized entry to data or functions.
* **Assume Identity:** Modifying another user's profile or account settings.
* **Data Breach:** Mass-scraping sensitive user information by automated iteration of keys.

## Likelihood of Exploit

**High**
IDOR/CWE-639 is one of the most common web vulnerabilities because it is easy to test for manually and often overlooked by developers who assume authentication is sufficient for authorization.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Tools trace the flow of user-controlled IDs into database lookup functions ("sinks") to see if an ownership check is missing. (Effectiveness: High).
* **Manual Testing (Penetration Testing):** The most reliable method. Log in as "User A," attempt to access "User B's" resource ID, and see if the server permits it.
* **Dynamic Analysis:** Using intercepting proxies (like Burp Suite) to modify ID values in real-time during requests.

## Potential Mitigations

### Architecture and Design

* **Verify Ownership for Every Request:** For every data access, the server must verify that the authenticated user has sufficient entitlements for that specific record.
* **Indirect Reference Maps:** Instead of exposing internal database keys, use temporary, per-session "maps" (e.g., User A sees their IDs as `1, 2, 3`, which the server maps internally to `9823, 4821, 1029`).
* **Non-Guessable Keys (UUIDs):** Use universally unique identifiers (UUIDs) instead of sequential integers to make it computationally difficult for an attacker to guess other valid keys.

### Implementation

* **Encryption and Digital Signatures:** Encrypt the keys sent to the client or attach a digital signature (HMAC) so the server can detect if the key has been tampered with.
* **Remove External Control:** Wherever possible, derive the key from the user's session data on the server side rather than accepting it as an input parameter.

## Simplified Example

```python
# VULNERABLE: The server trusts the 'account_id' from the URL
@app.route('/get_balance')
def get_balance():
    account_id = request.args.get('account_id') # User-controlled key
    # No check: Does this account belong to the logged-in user?
    balance = db.lookup_balance(account_id) 
    return jsonify(balance)

# SECURE: Validating ownership before retrieval
@app.route('/get_balance')
def get_balance():
    user_id = session.get('user_id') # Identity from trusted session
    account_id = request.args.get('account_id')
    
    # Verify that the user has access to this specific account_id
    if db.is_owner(user_id, account_id):
        balance = db.lookup_balance(account_id)
        return jsonify(balance)
    else:
        abort(403) # Forbidden

```
