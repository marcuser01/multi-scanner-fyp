# CWE-434: Unrestricted Upload of File with Dangerous Type

## Description

The product allows the upload or transfer of dangerous file types that are automatically processed within its environment. This occurs when an application accepts a file without sufficiently validating its type, extension, or contents, allowing an attacker to upload a file that the web server or operating system may execute as code.

## Why This Matters (Security Impact)

Unrestricted file uploads are often the shortest path to a total system compromise.

* **Remote Code Execution (RCE):** An attacker uploads a script (e.g., `.php`, `.jsp`, `.asp`) and then requests it via the browser, forcing the server to execute malicious code (a "web shell").
* **System Takeover:** Once a web shell is active, the attacker can browse the file system, access databases, or pivot to the internal network.
* **Cross-Site Scripting (XSS):** Uploading `.html` or `.svg` files containing malicious scripts that execute in the context of other users' sessions.

## Likelihood of Exploit

**Medium**
While common, successfully exploiting this often requires the attacker to know where the file is stored and for that directory to have "execute" permissions.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Tools identify upload "sinks" and check for the presence (or absence) of validation logic. (Effectiveness: High).
* **Manual Source Code Review:** Essential for verifying that validation isn't easily bypassed (e.g., checking for "double extensions" or case-sensitivity issues). (Effectiveness: High).
* **Dynamic Analysis (DAST):** Web scanners attempt to upload various executable extensions and then "call" them to see if they run. (Effectiveness: Partial).

## Potential Mitigations

### Architecture and Design

* **Unique Filenames:** Generate a new, unique filename for uploaded files instead of using the user-supplied name to prevent overwriting critical files or predictable execution paths.
* **Storage Outside Web Root:** Store uploaded files in a directory that is **not** accessible via a URL. Deliver them dynamically through a script that enforces access control.
* **Accept Known Good (Allowlist):** Define a strictly limited set of allowable extensions (e.g., `.jpg`, `.pdf`) and reject everything else.

### Implementation

* **Server-Side Validation:** Never rely on client-side checks (e.g., JavaScript validation in the browser), as these are easily bypassed by attackers using proxies.
* **Verify Extensions:** Ensure only **one** extension is used. Some servers (like Apache) may process `file.php.gif` as a PHP file.
* **Case-Insensitivity:** Perform case-insensitive checks (e.g., `.PHP` vs `.php`) on systems that treat them the same.
* **Content Inspection:** Do not rely solely on the MIME type or filename. Check the file's "magic numbers" (headers), but be aware that attackers can hide code in legitimate segments (like GIF comments).

### Environment Hardening

* **Least Privilege:** Run the web server with a low-privilege account to limit the damage if a file is executed.
* **Sandbox/Jail:** Use tools like **AppArmor**, **SELinux**, or **chroot** to restrict what an uploaded file can do if it manages to run.

## Simplified Example

```php
// VULNERABLE: Direct upload to web-accessible folder with user filename
$target_path = "uploads/" . $_FILES['userfile']['name'];
move_uploaded_data($_FILES['userfile']['tmp_name'], $target_path);

// SECURE: Rename file, check extension, and store outside web root
$allowed = ['jpg', 'png', 'pdf'];
$ext = pathinfo($_FILES['userfile']['name'], PATHINFO_EXTENSION);

if (in_array(strtolower($ext), $allowed)) {
    // Generate a unique ID to prevent filename-based attacks
    $safe_name = bin2hex(random_bytes(16)) . "." . $ext;
    $target_path = "/var/safe_storage/" . $safe_name; 
    move_uploaded_file($_FILES['userfile']['tmp_name'], $target_path);
}

```
