# CWE-121: Stack-based Buffer Overflow

## Description

A stack-based buffer overflow occurs when a buffer allocated on the execution stack (typically a local variable) is overwritten with more data than it can hold. Because the stack also contains critical security data—such as **return addresses** and **frame pointers**—writing beyond a buffer's boundary allows an attacker to hijack the program's control flow.

## Why This Matters (Security Impact)

Stack overflows are a classic high-impact vulnerability. By overwriting the stored return address, an attacker can:

* **Execute Arbitrary Code:** Redirect execution to a memory address containing malicious shellcode.
* **Return-into-libc:** Bypass "non-executable stack" protections by jumping directly to existing system routines (like `system()`).
* **Subvert Protections:** Manipulate stack pointers or frame pointers to create "write-what-where" primitives, subverting other security services.

## Common Consequences

* **Execute Unauthorized Code:** Running commands with the full privileges of the vulnerable program.
* **DoS: Crash or Resource Consumption:** Overwriting stack metadata usually leads to an immediate process crash or infinite loops.
* **Bypass Protection Mechanisms:** Neutralizing security services through control-flow hijacking.

## Background Details

The execution stack stores the **return address**, which tells the CPU where to continue once a function finishes. An attacker can overwrite this address with the location of their own code or a library function. Furthermore, modifying the **stack pointer** or **frame pointer** can be leveraged to compute incorrect memory addresses, facilitating further memory corruption.

## Likelihood of Exploit

**High**
Exploitation techniques (like "Smashing the Stack") are well-documented and highly reliable when modern protections are absent or misconfigured.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Tools find "sinks" like `gets()` or unbounded `strcpy()` calls that operate on stack variables. (Effectiveness: High).
* **Fuzzing:** Generating large or malformed inputs to trigger crashes that indicate stack corruption. (Effectiveness: High).
* **AddressSanitizer (ASan):** Integrated during compilation to catch runtime memory safety errors. (Effectiveness: Moderate).

## Potential Mitigations

### Environment Hardening

* **Stack Canaries:** Use compiler flags (e.g., `/GS` in Visual Studio, `StackGuard` in GCC) to insert "canary" values. If the canary is modified, the program exits before the return address is used.
* *Reference: D3-SFCV (Stack Frame Canary Validation).*


* **ASLR & PIE:** Randomize the positions of the executable and libraries in memory to make target addresses unpredictable.
* *Reference: D3-SAOR (Segment Address Offset Randomization).*


* **Non-Executable Stack (DEP/NX):** Ensure the stack segment is not executable, preventing the CPU from running code injected into a buffer.

### Implementation

* **Safe Functions:** Replace dangerous functions like `gets()` with bounded versions like `fgets()`.
* **Bounds Checking:** Explicitly perform size verification on all inputs before copying them into stack-allocated buffers.
* **Abstraction Libraries:** Use vetted libraries that abstract away risky low-level APIs.

## Simplified Example

```c
// VULNERABLE: 'buffer' is on the stack; input is not checked
void vulnerable_func(char *input) {
    char buffer[64];
    strcpy(buffer, input); // Overflow occurs if input > 64 chars
}

// SECURE: Boundary is enforced
void secure_func(char *input) {
    char buffer[64];
    strncpy(buffer, input, sizeof(buffer) - 1);
    buffer[63] = '\0'; // Ensure null-termination
}

```
