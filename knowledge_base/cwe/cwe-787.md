# CWE-787: Out-of-bounds Write

## Description

The product writes data past the end or before the beginning of the intended buffer. This typically occurs in languages like C and C++ that do not perform automatic bounds checking. It is a specific, dangerous subset of buffer overflows that often leads to memory corruption.

## Why This Matters (Security Impact)

An out-of-bounds write is one of the most critical software weaknesses because it allows:

* **Arbitrary Code Execution (ACE):** Attackers can overwrite sensitive control data, such as return addresses on the stack or function pointers on the heap, to redirect execution flow.
* **Privilege Escalation:** Overwriting local variables that manage authentication or authorization status.
* **System Instability:** Corrupting the process's data structures, leading to crashes or unpredictable behavior.

## Common Consequences

* **Modify Memory:** Intentional corruption of program memory to alter logic.
* **Execute Unauthorized Code:** Redirecting execution to attacker-controlled payloads.
* **Denial of Service (DoS):** Triggering application crashes by accessing invalid memory addresses.

## Likelihood of Exploit

**High**
Memory corruption vulnerabilities are highly sought after by advanced attackers. While modern OS protections exist, the impact of a successful exploit is total system compromise.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Modern tools use data flow analysis and constraint-checking to find potential overflows. (Effectiveness: High).
* **Automated Dynamic Analysis (Fuzzing):** Stress-testing functions with diverse, large inputs to trigger memory faults. (Effectiveness: Moderate).
* **Runtime Sanitizers:** Using tools like **AddressSanitizer (ASan)** during development to detect out-of-bounds access as it happens. (Effectiveness: High).

## Potential Mitigations

* **Language Selection:** Use memory-safe languages (Java, C#, Rust, Python) that handle bounds checking automatically.
* **Safe Libraries:** Use safer string and memory handling libraries such as **Strsafe.h** (Windows) or **SafeStr**.
* **Compiler Defenses:** Enable stack canaries (e.g., `/GS` flag or `StackGuard`) to detect stack-based overflows before they are exploited.
* **Environment Hardening:**
* **ASLR (Address Space Layout Randomization):** Randomize memory addresses to make jumps to malicious code unpredictable.
* **DEP/NX (Data Execution Prevention):** Mark data segments (stack/heap) as non-executable to prevent payload execution.


* **Boundary Validation:** Always check index and length calculations in loops. Use bounded functions like `strncpy()` instead of `strcpy()`.
* **Input Truncation:** Sanitize and truncate input lengths before passing them to memory-copy functions.

## Simplified Example

```c
// VULNERABLE: No check on source length relative to destination
void manipulate(char *source) {
    char dest[10];
    strcpy(dest, source); // Can write past 10 bytes
}

// SECURE: Length is restricted to the destination buffer size
void manipulate(char *source) {
    char dest[10];
    strncpy(dest, source, sizeof(dest) - 1);
    dest[sizeof(dest) - 1] = '\0'; // Ensure null-termination
}

```
