# CWE-122: Heap-based Buffer Overflow

## Description

A heap-based buffer overflow occurs when a buffer allocated in the heap portion of memory is overwritten. This typically involves memory allocated using routines such as `malloc()`, `calloc()`, or the `new` operator in C++. Because the heap is used for dynamic memory allocation, these overflows often corrupt metadata used by the memory manager or adjacent data objects.

## Why This Matters (Security Impact)

Heap overflows are highly exploitable and can bypass many standard protections that focus on the stack.

* **Execute Arbitrary Code:** Attackers can overwrite **function pointers** that reside in heap memory. When the application later calls that pointer, it executes the attacker's code.
* **Corrupt C++ Objects:** Object methods in C++ are often implemented using virtual method tables (vtable pointers). Overwriting these pointers allows for redirection of program logic.
* **Global Offset Table (GOT) Manipulation:** In C programs, the underlying runtime often uses a GOT which can be targeted to subvert security services.
* **Subversion of Security Services:** Successful arbitrary code execution can be used to disable or bypass any other security service within the application.

## Common Consequences

* **Execute Unauthorized Code:** Redirecting execution via corrupted function pointers or vtables.
* **Modify Memory:** Overwriting critical user data or system metadata.
* **Denial of Service (DoS):** Triggering crashes, segmentation faults, or infinite loops that consume CPU and memory resources.
* **Bypass Protection Mechanism:** Intel gathered from heap layout can be used to defeat defenses.

## Likelihood of Exploit

**High**
While more complex than stack overflows, heap overflows are a primary target for sophisticated exploits due to the wealth of function pointers and metadata available in dynamic memory.

## How This Is Typically Detected

* **Fuzzing:** Generating diverse, algorithmic inputs to trigger unexpected results like memory corruption or crashes. (Effectiveness: High).
* **Automated Dynamic Analysis (ASan):** Using **AddressSanitizer** during compilation to insert runtime checks that report the exact point of memory safety violations. (Effectiveness: Moderate).
* **Manual Code Review:** Identifying risky manual memory management patterns and the use of unbounded copy functions on heap-allocated buffers.

## Potential Mitigations

### Architecture and Design

* **Language Selection:** Use languages that perform automatic bounds checking or provide safe memory abstractions.
* **Abstraction Libraries:** Use vetted libraries to abstract away risky, low-level APIs.

### Environment Hardening

* **ASLR & PIE:** Randomize the memory addresses of the executable and libraries to prevent attackers from reliably jumping to malicious code.
* *Reference: D3-SAOR (Segment Address Offset Randomization).*


* **Automatic Detection:** Use compiler flags like `/GS` (Visual Studio) or `FORTIFY_SOURCE` (GCC) for range and index checking.
* **OS-Level Prevention:** Enable OS-level features like non-executable memory pages to prevent code execution from data segments.

### Implementation

* **Bounds Checking:** Always perform explicit checks on input size before copying data into heap buffers.
* **Safe Functions:** Replace dangerous functions like `gets()` with safe equivalents that enforce boundary limits.

## Simplified Example

```c
// VULNERABLE: Input is copied into heap memory without size validation
void handle_data(char *input) {
    char *heap_buffer = (char *)malloc(32);
    strcpy(heap_buffer, input); // Overflow if input > 32 bytes
    // ...
}

// SECURE: Boundary is verified before the copy operation
void handle_data(char *input) {
    char *heap_buffer = (char *)malloc(32);
    if (heap_buffer != NULL) {
        strncpy(heap_buffer, input, 31);
        heap_buffer[31] = '\0'; // Ensure termination
    }
}

```
