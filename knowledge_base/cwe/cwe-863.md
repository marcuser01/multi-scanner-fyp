# CWE-863: Incorrect Authorization

## Description

The product performs an authorization check when an actor attempts to access a resource or perform an action, but it does not correctly perform the check. While **Authentication** verifies *who* a user is, **Authorization** determines *what* they are allowed to do. This weakness occurs when the logic governing these permissions is flawed, bypassed, or insufficient to protect sensitive data and functionality.

## Why This Matters (Security Impact)

Incorrect authorization is a "High" likelihood exploit that can lead to total system compromise.

* **Privilege Escalation:** An attacker with "normal" user access can perform "administrative" actions, such as deleting users or changing system configurations.
* **Data Exfiltration:** Attackers can read sensitive files, directories, or database records that should be restricted to specific roles or owners.
* **Insecure Direct Object Reference (IDOR):** A common subset where a user accesses another user's data by simply changing an ID in a URL or request because the server fails to verify ownership.
* **Denial of Service (DoS):** Unauthorized users might gain access to resource-intensive functions, excessively consuming CPU or memory to crash the system.

## Common Consequences

* **Read/Modify Application Data:** Unauthorized access to sensitive data stores or files.
* **Gain Privileges:** Assuming the identity or rights of a more privileged user.
* **Execute Unauthorized Code:** Using elevated privileges to run commands or scripts.
* **Bypass Protection Mechanisms:** Circumventing the intended security boundaries of the application.

## Likelihood of Exploit

**High**
Authorization logic is often custom-built and complex, making it a frequent target for manual testing and automated exploitation.

## How This Is Typically Detected

* **Manual Analysis (Penetration Testing):** This is the most effective method. Humans can understand business logic and test if a "User A" can access "User B's" data. (Effectiveness: Moderate to High).
* **Architecture/Design Review:** Identifying missing checks at the design phase. (Effectiveness: High).
* **Automated Static Analysis (SAST):** Can detect missing checks in known frameworks or misconfigured access files (like `.htaccess`), but often struggles with custom business logic. (Effectiveness: Limited).
* **Dynamic Analysis (DAST):** Scanners can find unprotected endpoints, but rarely understand the nuances of role-based access. (Effectiveness: Partial).

## Potential Mitigations

### Architecture and Design

* **Role-Based Access Control (RBAC):** Divide the product into anonymous, normal, and administrative areas. Map roles strictly to functionality.
* **Default Deny:** Configure the environment to deny all access by default, only granting permissions explicitly.
* **Business Logic Alignment:** Ensure checks aren't just for generic resources (like files) but also for business rules (e.g., a doctor can only see *their* patients' records).
* **Vetted Frameworks:** Use established libraries like **JAAS** or **OWASP ESAPI** rather than building custom authorization logic from scratch.

### Implementation

* **Server-Side Enforcement:** Always perform checks on the server. Never rely on hiding a button in the UI or client-side checks, as attackers can bypass the UI entirely.
* **Session Validation:** Every sensitive request must be accompanied by an active, authenticated session token that is verified against the required permissions for that specific resource.
* **Disable Caching:** Ensure pages with sensitive data are not cached by the browser to prevent unauthorized local access.

## Simplified Example

```python
# VULNERABLE: Only checks if the user is logged in, not if they OWN the record
@app.route('/view_invoice/<invoice_id>')
def view_invoice(invoice_id):
    if user_is_authenticated():
        return database.get_invoice(invoice_id) # IDOR: User 1 can see User 2's invoice

# SECURE: Checks authentication AND authorization (ownership)
@app.route('/view_invoice/<invoice_id>')
def view_invoice(invoice_id):
    user = get_current_user()
    invoice = database.get_invoice(invoice_id)
    
    if user and invoice and (invoice.owner_id == user.id or user.is_admin):
        return invoice
    else:
        abort(403) # Forbidden

```
