# CWE-862: Missing Authorization

## Description

The product fails to perform an authorization check when an actor attempts to access a resource or perform an action. While the user may be authenticated (identified), the system does not verify if they have the actual permission to access a specific object or execute a specific function.

## Why This Matters (Security Impact)

Missing authorization is a critical failure in the access control layer. It allows:

* **Horizontal Privilege Escalation:** An authenticated user accessing data belonging to other users of the same rank (e.g., viewing another user's private profile).
* **Vertical Privilege Escalation:** A standard user accessing administrative endpoints or restricted system functions.
* **Direct Object Access:** Accessing sensitive files or database records by guessing a URL or ID (Insecure Direct Object Reference).

## Common Consequences

* **Read/Modify Application Data:** Attackers can view or alter sensitive data store contents without restriction.
* **Bypass Protection Mechanism:** Privileged functionality becomes accessible to unauthorized actors.
* **Denial of Service (DoS):** Unauthorized users can trigger resource-heavy tasks, leading to system exhaustion.

## Likelihood of Exploit

**High**
Exploitation is often trivial (e.g., changing a numeric ID in a URL) and does not require advanced technical skills once an interface is discovered.

## How This Is Typically Detected

* **Manual Analysis:** Penetration testing and threat modeling are the most effective ways to identify missing logic checks. (Effectiveness: Moderate/High).
* **Automated Dynamic Analysis (DAST):** Scanners can identify endpoints that do not require valid session tokens or permissions. (Effectiveness: Moderate).
* **Automated Static Analysis (SAST):** Useful for identifying missing annotations or library calls in standard frameworks (e.g., Spring Security), but struggles with custom logic. (Effectiveness: Limited).

## Potential Mitigations

* **Default Deny Policy:** Restrict all access by default. Explicitly grant access to specific roles or users for every endpoint and resource.
* **Role-Based Access Control (RBAC):** Divide the application into anonymous, normal, and administrative areas. Map roles to specific data and functions.
* **Server-Side Enforcement:** Always perform authorization checks on the server. Never rely on client-side UI elements (e.g., hidden buttons) to prevent access.
* **Framework Integration:** Utilize vetted authorization frameworks like **Spring Security**, **JAAS**, or **OWASP ESAPI Access Control**.
* **Business Logic Checks:** Ensure checks are specific to the record being accessed (e.g., verifying that a `patient_id` matches the authenticated `doctor_id`).
* **Disable Caching:** Prevent sensitive, authorized pages from being stored in local browser caches.

## Simplified Example

```python
# VULNERABLE: Only checks if user is logged in, not if they own the record
@app.route('/view_invoice/<invoice_id>')
def view_invoice(invoice_id):
    if not current_user.is_authenticated:
        return redirect(url_for('login'))
    return db.get_invoice(invoice_id)

# SECURE: Checks identity AND specific authorization for the resource
@app.route('/view_invoice/<invoice_id>')
def view_invoice(invoice_id):
    invoice = db.get_invoice(invoice_id)
    if not current_user.is_authenticated or (invoice.owner_id != current_user.id):
        abort(403) # Forbidden
    return invoice

```
