# CWE-125: Out-of-bounds Read

## Description

The product reads data past the end or before the beginning of the intended buffer. This occurs when a pointer or index is incremented or decremented beyond the boundary of the allocated memory block. Unlike an Out-of-bounds Write, which corrupts memory, an Out-of-bounds Read typically results in the disclosure of sensitive information or application crashes.

## Why This Matters (Security Impact)

While often considered less severe than "writes," Out-of-bounds Reads are critical precursors to advanced exploits:

* **Information Leakage:** Attackers can read sensitive data such as **cryptographic keys**, session tokens, or Personally Identifiable Information (PII) residing in adjacent memory.
* **ASLR Bypass:** By reading memory addresses, attackers can determine the layout of the program's memory, effectively neutralizing Address Space Layout Randomization (ASLR).
* **Exploit Chaining:** Discovered memory offsets are frequently used to improve the reliability of a separate code execution vulnerability.

## Common Consequences

* **Read Memory:** Unauthorized access to secret values or memory addresses (Confidentiality loss).
* **Bypass Protection Mechanism:** Gathering intel to defeat security mitigations like ASLR.
* **Denial of Service (DoS):** Accessing unmapped or protected memory addresses, causing a segmentation fault or system crash.

## Likelihood of Exploit

**High**
Out-of-bounds reads (such as the famous "Heartbleed" vulnerability) are highly reliable to exploit and often leave fewer traces than memory-corruption attacks.

## How This Is Typically Detected

* **Automated Static Analysis (SAST):** Tools like **Semgrep** or specialized C/C++ analyzers identify loops or pointer arithmetic that lack explicit boundary checks. (Effectiveness: High).
* **Fuzzing:** Using randomized inputs to trigger crashes or behavioral anomalies that indicate illegal memory access. (Effectiveness: High).
* **Automated Dynamic Analysis (ASan):** Using **AddressSanitizer** during development to detect and log "illegal read" operations at the exact moment they occur. (Effectiveness: Moderate/High).

## Potential Mitigations

* **Language Selection:** Use memory-safe languages (Java, Rust, Go, Python) that perform automatic bounds checking on all array/buffer accesses.
* **Input Validation:** Strictly validate any length, offset, or size arguments provided by untrusted sources before using them in memory operations.
* **Avoid Sentinel Reliance:** Do not rely solely on special characters (like a `NUL` terminator) in untrusted input to stop a read operation; always use explicit length limits.
* **Vetted APIs:** Use functions that require a buffer length (e.g., `memcpy_s` instead of `memcpy`) and ensure calculations for those lengths are correct.
* **Memory Abstractions:** Utilize higher-level data structures (like `std::vector` in C++ or `string` classes) that manage boundaries internally.

## Simplified Example

```c
// VULNERABLE: Relies on index from user without checking buffer size
int getValue(int *array, int index) {
    return array[index]; // If index > array size, an Out-of-bounds Read occurs
}

// SECURE: Validates index against the known size of the buffer
int getValue(int *array, int size, int index) {
    if (index < 0 || index >= size) {
        return -1; // Handle error: Index is out of bounds
    }
    return array[index];
}

```
