# CWE-352: Cross-Site Request Forgery (CSRF)

## Description

The web application fails to verify whether a request was intentionally initiated by the user. An attacker can trick a victim’s browser into sending an unauthorized request to a vulnerable application where the victim is currently authenticated. Because the browser automatically includes session cookies, the server treats the malicious request as legitimate.

## Why This Matters (Security Impact)

CSRF allows an attacker to perform actions as the victim without their knowledge.

* **Privilege Escalation:** If the victim is an administrator, the attacker can delete data, create new admin accounts, or shut down services.
* **Account Takeover:** Attackers can change the victim's password or email address via a forged request.
* **State Change:** Forging transactions, such as transferring funds or modifying profile settings.

## Common Consequences

* **Gain Privileges:** Performing administrative actions via the victim's session.
* **Modify Application Data:** Unauthorized changes to records, settings, or user profiles.
* **Non-Repudiation:** Actions performed by the attacker appear in logs as legitimate actions by the victim.

## Likelihood of Exploit

**Medium**
While widely understood, successful exploitation requires the victim to have an active session and visit a malicious site or link simultaneously.

## How This Is Typically Detected

* **Manual Analysis:** Penetration testing and session recording are highly effective for identifying state-changing requests lacking protection. (Effectiveness: High).
* **Dynamic Analysis (DAST):** Web application scanners can identify forms or endpoints that do not utilize anti-forgery tokens. (Effectiveness: High).
* **Static Analysis (SAST):** Generally difficult to detect reliably via pure static analysis as it depends on business logic and implicit security policies. (Effectiveness: Limited).

## Potential Mitigations

* **Anti-CSRF Nonces (Synchronizer Token):** Generate a unique, cryptographically strong, and unpredictable nonce for each user session or form. Verify this token on the server for every state-changing request.
* **Double-Submit Cookies:** Send a random value in both a cookie and a request parameter. The server validates that both values match.
* **SameSite Cookie Attribute:** Use the `SameSite=Strict` or `SameSite=Lax` attribute on session cookies to prevent the browser from sending them during cross-site requests.
* **Avoid GET for State Changes:** Ensure that `GET` requests are idempotent (read-only) and do not trigger any side effects or data modifications.
* **Custom Request Headers:** Require custom HTTP headers (e.g., `X-Requested-With`) for API calls, which are protected by the Same-Origin Policy.
* **Mandatory XSS Protection:** Remediate all **CWE-79 (XSS)** vulnerabilities first; an XSS flaw can be used to read anti-CSRF tokens and bypass these defenses.

## Simplified Example

```html
<form action="/transfer-funds" method="POST">
    <input type="text" name="amount">
    <input type="submit">
</form>

<form action="/transfer-funds" method="POST">
    <input type="hidden" name="csrf_token" value="a7b9c3d1e4f2g5h6">
    <input type="text" name="amount">
    <input type="submit">
</form>

```
